# Databricks 習熟度向上のためのスプリント・バックログ（最終版）

Databricksの各コンポーネント（データエンジニアリング、データサイエンス＆ML、データウェアハウス/BI、データガバナンス）を網羅的に体験し、実践的なデータ運用スキルを向上させるためのプロジェクト・バックログです。

## 📌 Epic 1: パイプラインの仕上げと自動化

*目的: 既存のBronze/Silver/Goldパイプラインの問題を修正し、日次自動実行を実現する*

| Story | 内容 |
|-------|------|
| **冪等性の確保** | |
| 1-1 | Bronze取り込みを冪等にする（`mode("overwrite")`への変更、またはMERGE文の導入）。同じ処理を何回実行しても結果が同じになる状態を作る |
| 1-2 | Silver/Gold変換も同様に冪等にする |
| **自動化** | |
| 1-3 | Databricks Workflows (Job) を設定し、Bronze→Silver→Goldを日次自動実行するスケジュールを組む |
| 1-4 | Job実行時のCompute設定を**Job Cluster**に切り替える（All-Purpose比で約30%コスト削減） |
| 1-5 | Job実行結果の成功/失敗をメール通知する設定を行う |
| **コスト最適化** | |
| 1-6 | 【要フィージビリティ確認】NAT Gatewayの時間制限運用を検討する（パイプライン実行時のみNAT Gatewayを作成し、終了後に削除するLambda/CloudFormation構成） |

## 📌 Epic 2: データガバナンスとアクセス制御の運用リハーサル

*目的: 「誰が」「どの層/テーブル/カラム」にアクセスできるかを設計・実装・証明する*
*※ できるところからEpic 1と並行して着手*

| Story | 内容 |
|-------|------|
| **設計** | |
| 2-1 | ペルソナ設計書を作成する。3ロールを定義：`analyst_user`（Gold層のみ）、`marketing_user`（特定ビューのみ）、`data_engineer`（全層フルアクセス） |
| **実装** | |
| 2-2 | Unity Catalogで上記ロールに対応するグループを作成し、GRANT/REVOKE文でスキーマ・テーブル単位の権限を付与する |
| 2-3 | 従業員テーブルの個人情報（電話番号、住所）をマスキングする動的ビューを作成する（`is_member()`関数を使用） |
| 2-4 | marketing_user向けに、売上サマリビューのみSELECT可能な細粒度権限を設定する |
| **証明テスト** | |
| 2-5 | analyst_userでログインし、Bronze/Silver層にアクセスすると`PermissionDenied`になることを証明する |
| 2-6 | marketing_userでログインし、許可されたビュー以外のGoldテーブルにアクセスできないことを証明する |
| 2-7 | HR以外のユーザーが従業員ビューを見た時、個人情報が`***`と表示されることを証明する |
| 2-8 | 全テスト結果をスクリーンショット＋assertで証跡として記録する |

| 6-4 | 従業員管理機能を実装する（従業員情報の閲覧・編集、権限に応じたマスキング表示） |
| 6-5 | 業務アプリの入力バリデーションを実装する（必須項目チェック、数値範囲チェック等） |
| **ユーザー側アプリ（購買画面）** | |
| 6-6 | Streamlit等でユーザー向け購買アプリの骨格を作成する（商品一覧画面） |

## 📌 Epic 7: テスト戦略の実装（単体テスト・結合テスト・E2Eテスト）

*目的: Epic 6までに構築した全システムに対して、体系的なテストを実装する。スタブの概念と使い分けを体験する*

| Story | 内容 |
|-------|------|
| **単体テスト（スタブ使用）** | |
| 7-1 | Silver変換ロジックの単体テストを作成する。RDSやBronze層に依存せず、**固定データ（スタブ）をPythonコード内で生成**し、変換結果が期待通りかassertする |
| 7-2 | Gold集計ロジックの単体テストを作成する。Silver層の代わりに**スタブDataFrame**を入力し、集計結果の数値が期待値と一致するかassertする |
| 7-3 | 業務アプリのRDS書き込みロジックの単体テストを作成する。実際のRDSではなく**スタブ（モックDB）**を使い、INSERT文が正しく生成されるかassertする |
| 7-4 | 購買アプリの在庫チェックロジックの単体テストを作成する。在庫0の商品に対する注文が正しくエラーになるかassertする |
| **結合テスト（一部スタブ）** | |
| 7-5 | Bronze→Silver→Goldの3層を通しで実行する結合テストを作成する。入力はRDSではなく**テスト用のBronze Delta Table（スタブ）**を事前に作成して使う |
| 7-6 | 異常系テスト：不正データ（NULL、マイナス金額、重複キー等）をスタブに混入させ、Silver層のDQチェックで正しく検知・除外されることを確認する |
| 7-7 | アプリ→RDS書き込み→Bronze取り込みの結合テスト。アプリ部分は本物、パイプライン部分も本物だが、**テスト専用のRDSスキーマ**を使って本番データを汚さない |
| **E2Eテスト（スタブなし・全部本物）** | |
| 7-8 | 購買アプリで注文 → RDS更新 → パイプライン実行 → Gold層の集計結果が期待値と一致 → ダッシュボードに反映。**全て本物のシステムを使う** |
| 7-9 | 業務アプリで商品登録 → RDS更新 → パイプライン実行 → Gold層に反映 → ダッシュボードに表示。**全て本物** |
| 7-10 | E2Eテスト後のクリーンアップ処理を実装する（テストデータの削除、テーブルの復元） |
| **テスト自動実行** | |
| 7-11 | 上記テストをDatabricks Workflowのタスクとして登録し、日次または週次で自動実行されるようにする |
| 7-12 | テスト結果を`ops.test_results`テーブルに記録し、テスト履歴ダッシュボードを作成する |
| 7-13 | テスト失敗時にメール通知が飛ぶように設定する |

## 📌 Epic 8: データ品質（DQ・DLT）とストリーミング

*目的: 品質保証を経常運用に組み込み、DLT Expectationsによる宣言的品質管理とストリーミング処理を体験する*
*※ Story 8-1〜8-4はリスクが高いためSprint 1で先行実施する*

| Story | 内容 |
|-------|------|
| **既存データの品質保証（Sprint 1で先行実施）** | |
| 8-1 | 現在のBronze層全テーブルについて、重複レコードの有無を調査する。`_load_date`別の件数と、主キー別のユニーク件数を比較し、2回実行による重複がないか確認する |
| 8-2 | 重複が発見された場合、重複を除去するクリーンアップ処理を実行する（最新の`_load_date`のみ残す、またはDISTINCTで洗い替え） |
| 8-3 | Silver層・Gold層についても同様の重複調査・クリーンアップを実施する |
| 8-4 | 上記の重複チェックを**経常的に実行するDQチェックジョブ**として登録する（日次パイプライン実行後に自動で走る品質ゲート） |
| **DLTによる宣言的品質管理** | |
| 8-5 | 既存のBronze→Silver→Gold処理をDelta Live Tables (DLT)で書き直す |
| 8-6 | DLT Expectationsでデータ品質制約を宣言的に定義する（例: `EXPECT (unit_price >= 0)`, `EXPECT (order_id IS NOT NULL)`, `EXPECT (重複なし)`） |
| 8-7 | 意図的に不正データを投入し、Expectationsが検知・除外する様子をDLTのUIで確認する |
| **ストリーミング** | |
| 8-8 | Auto Loader / Structured Streamingで、S3上のダミーログファイルを継続的に取り込むストリーミング処理を構築する |
| **比較・ドキュメント** | |
| 8-9 | Epic 7で作った手動テスト/DQチェック、Story 8-4の経常DQジョブ、DLT Expectationsの3つを比較し、それぞれの長所・短所をドキュメントにまとめる |
