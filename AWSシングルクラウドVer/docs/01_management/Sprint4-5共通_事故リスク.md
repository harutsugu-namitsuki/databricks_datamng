# Sprint 4・5 共通 事故リスク

Sprint 4（ML構築）と Sprint 5（BI ダッシュボード）の両方の実施手順書・コードに共通して存在する、
**手順書通りに進めたら "ほぼ確実に事故る" レベルのリスク** をまとめたファイルです。

---

## 🔥 事故1: カタログ名 `northwind_catalog` のハードコード

**【影響範囲】** Sprint 4 の全4ノートブック（`.py`）＋ Sprint 5 の全8クエリ（`.sql`）＝ **合計12ファイル**
**【事故の内容】**
すべてのノートブックおよびSQLファイルで、`northwind_catalog` というカタログ名がリテラル文字列として直接書かれています。
もし環境構築時にカタログ名を変更していた場合（例：`dev_catalog`, `test_northwind` 等）、**全ファイルが `TABLE_OR_VIEW_NOT_FOUND` で即死** します。

**【なぜ事故と言えるか】**
- 開発環境・検証環境・本番環境でカタログ名が異なることはよくあるパターン
- Sprint 4 では Python 変数 `CATALOG_NAME` に入れてはいるが、SQL ファイル（Sprint 5）ではそれすらなく FROM 句に直書き
- 12ファイルを1つずつ手で直すのは面倒で、漏れが出やすい

**【予防策】**
- 作業開始前に、自分の環境のカタログ名と手順書の `northwind_catalog` が一致しているか必ず確認する
- 一致しない場合は、全ファイルを一括置換する（エディタの「ファイル横断検索・置換」等を使用）

## 🔥 事故2: Databricks UI の世代ギャップ（手順書が旧UIを前提にしている）

**【影響範囲】** Sprint 4 の Step 1-1 および Sprint 5 の Phase 3・5（ダッシュボード作成手順全体）
**【事故の内容】**
Databricks は頻繁にUIをアップデートしており、手順書が作成された時点と現在のUIでは、**メニュー名・画面遷移・デフォルト挙動が異なる** ケースがある。具体的には：

| Sprint | 手順書の記述 | 現在のDatabricksの実態 |
|--------|------------|---------------------|
| Sprint 4 | 「Standard Runtimeでも AutoML は動作する」 | **ML Runtime が必須**。Standard では `ImportError` で即死 |
| Sprint 5 | 「ダッシュボードを作成」→ クエリの可視化をウィジェットとして追加 | ← これは **Classic Dashboard** の手順。現在のデフォルトは **AI/BI Dashboard（旧 Lakeview）** であり、UI操作が全く異なる |

**【なぜ事故と言えるか】**
- 手順書通りにボタンを押しても、画面に出てくるものが手順書と違うため、作業者が「何か間違えた？」とパニックになる
- 特にDatabricks初学者や、手順書を信頼して進める作業者ほど詰まる
- MLランタイムの件は「動く」と書いてあるのに動かない、という最悪のパターン

**【予防策】**
- Sprint 4: クラスター作成時に **必ず Databricks Runtime ML** を選択する（Standard は不可）
- Sprint 5: ダッシュボード作成時に **Classic Dashboard** を明示的に選択する（またはAI/BIダッシュボードの操作方法に読み替える）
