# Databricks 習熟度向上のためのスプリント・バックログ

Databricksの各コンポーネント（データエンジニアリング、データサイエンス＆ML、データウェアハウス/BI、データガバナンス）を網羅的に体験し、実践的なデータ運用スキルを向上させるためのプロジェクト・バックログです。

## 💡 ユーザーアイデアの精査（ITコンサルタント視点）

まず、現時点で想定されている9つのアイデアについて、「**Databricksの習熟度向上にどう寄与するか**」と「**実現可能性**」の観点で評価・精査します。

| No. | アイデア | 評価・フィードバック |
| :--- | :--- | :--- |
| **①** | RDSからDatabricksへ1日1アップロードすることで、何かメリットはないか。 | **【◎ 大いにあり / 実現可能】**<br>分析用DB(Databricks/OLAP)と業務システムDB(RDS/OLTP)を分けるベストプラクティスを学べます。Databricks側で過去から現在までのスナップショット（履歴データ）を保持・分析できるのが最大のメリットです。 |
| **②** | その際の金の試算 | **【◎ 必須スキル / 実現可能】**<br>Databricksの「System Tables（課金テーブル）」を用いたコスト分析ダッシュボード作成により、FinOps（クラウドコスト最適化）のスキルが身につきます。 |
| **③** | RDSが定められた時間帯しか動かないとかできるか。 | **【〇 / 実現可能】**<br>Databricks自体の機能というよりAWS側の機能（EventBridge + Lambdaなど）になりますが、Databricks JobからAPIを叩いてRDSを起動し、処理が終われば停止するといった高度なインフラ制御・コスト削減策も可能です。 |

| **⑤** | AI/MLを作って、保存。 | **【◎ 非常に重要 / 実現可能】**<br>Databricksのコア機能である「**MLflow**」を利用し、機械学習モデルの実験追跡や、モデルレジストリによるバージョン管理が学べます。データサイエンティストとしての強力な経験になります。 |

## 🏃‍♂️ スプリント・バックログ（Epic & Story）

上記のアイデア群を、システム開発のアジャイル手法（スクラム）になぞらえ、Epic（大きなテーマ）と Story（具体的なタスク）として整理しました。この順番でスプリント（反復開発）を回すことで、段階的かつ確実に習熟度が上がります。

### 📌 Epic 1: データパイプラインの基礎構築 (Data Engineering)
*目的: RDSのデータをDatabricksのデータレイクハウスに統合し、分析可能な土台を作る（アイデア①, ⑥）*

* **Story 1-1**: AWS RDS (Northwind) と Databricks をJDBC等で接続し、テーブルデータを抽出する Notebook を作成する。
* **Story 1-2**: 抽出したデータを、Bronze（生データ）、Silver（クレンジング・結合）、Gold（集計済み）の 3層アーキテクチャ（メダリオンアーキテクチャ）で Delta Table として保存する。
* **Story 1-3**: Databricks Workflows (Job) を設定し、1日1回（日次バッチ）RDSの最新データをDatabricksに取り込むスケジュール処理を自動化する。

### 📌 Epic 2: ダッシュボードとコスト管理 (Data Analyst & FinOps)
*目的: 蓄積したデータを可視化し、レポーティングとコスト管理を自動化する（アイデア②, ④, ⑧）*

* **Story 2-1**: Databricks SQL を用いて、Northwind の売上や在庫状況を可視化する BI ダッシュボードを作成する。
* **Story 2-2**: ダッシュボードのスケジュール配信機能を設定し、毎週月曜の朝9時に最新の売上レポートをメール（またはPDF）で自動配信する。
* **Story 2-3**: Unity Catalog の System Tables をクエリし、Databricksクラスタなどの利用料金をモニタリングするコスト試算ダッシュボードを作成する。

### 📌 Epic 3: データガバナンスとセキュリティ (Data Governance)
*目的: 「誰が」「どの」データにアクセスできるかを厳密に管理する（アイデア⑦）*

* **Story 3-1**: Unity Catalog 管理下で新しいテストユーザー/グループ（例: `Analyst`, `Marketing`）を想定して権限モデルを設計する。
* **Story 3-2**: 特定のユーザーには Northwind の「従業員テーブル」の「給与や個人情報カラム」をマスキングするダイナミックビュー（列・行レベルセキュリティ）を実装する。
* **Story 3-3**: 実際の権限付与のテストを行い、意図したデータしか見えないことを証明する。

### 📌 Epic 4: 業務アプリとデータレイクハウスの統合 (End-to-End System)
*目的: システムの入り口（アプリ）から出口（分析基盤）までを一気通貫で構築・最適化する（アイデア③, ⑨）*

* **Story 4-1**: Streamlit（Python UI）などの簡易アプリを作成し、利用者が直接 RDS の Northwind データを更新・追加（商品の発注など）できるようにする。
* **Story 4-2**: アプリでの操作（データの更新）がトリガーとなり、または1日1回のパイプラインを通じて、変更内容が Databricks 側に確実に反映・同期されていることをテストする。
* **Story 4-3**: 【発展】コスト最適化のため、Databricks Jobの実行開始前にAWS LambdaやAPI経由でRDSを起動し、Job終了後にRDSを自動停止させるオーケストレーションを組み込む。

### 📌 Epic 5: 機械学習モデルの構築と管理 (Data Science)
*目的: 蓄積されたビジネスデータから未来の予測を行い、運用に乗せる（アイデア⑤）*

* **Story 5-1**: Databricks Machine Learning の Notebook を使い、Northwind の過去データから「商品別売上予測モデル」などの機械学習モデルを作成する。
* **Story 5-2**: MLflow を用いて、実験のパラメータ、精度メトリクス、及びモデルアーティファクトをトラッキング（記録）する。
* **Story 5-3**: 作成したモデルを MLflow Model Registry (Unity Catalog) に登録し、「Staging (検証)」「Production (本番)」というモデルのライフサイクル管理を体験する。
