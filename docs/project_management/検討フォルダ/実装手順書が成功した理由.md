# 実装手順書 vs Gemini版：正誤分析

## 結論：実装手順書が正しく、Gemini版が誤っている主要ポイント

---

## 🔴 Gemini版の重大な誤り

### 1. **IAMロールの扱い：最大の問題**

**実装手順書（正しい）：**
- CloudFormationで **2種類のIAMロールを事前作成**
  - **ワークスペースロール**（EC2管理用）→ Credentialに登録
  - **カタログロール**（Unity Catalog S3アクセス用）→ Storage / Storage Credentialに登録
- IAMロールの信頼ポリシー・権限ポリシーはすべてCloudFormationで定義済み

**Gemini版（誤り）：**
- Credentialの登録時に **IAMロールを手動でAWSコンソールから作成させている**（Step 2-2①の手順4〜15）
- CloudFormationで既に作成済みのロールがあるのに、それを無視して手動で別ロールを作らせている
- `databricks_workspace_policy.json` という外部ファイルを手動で貼り付けさせる手順になっている

> **問題点**: CloudFormationのOutputに `WorkspaceRoleArn` が出力されるのだから、それをそのまま貼るだけでよい。手動でロールを作る必要はまったくない。

---

### 2. **S3バケットポリシーの扱い**

**実装手順書（正しい）：**
> 「バケットポリシーはCloudFormationで事前設定済みのため手動作業不要です。」

**Gemini版（誤り）：**
- Storage登録時に「ポリシーを生成」ボタンを押してJSONを取得し、**手動でS3バケットポリシーを上書きさせている**（Step 2-2②の手順3〜4）
- 「これをやらないと後でjsonコードが自動生成されなくなります」と**誤った緊急感**を煽っている

> **問題点**: CloudFormationテンプレートにバケットポリシーが含まれているなら、手動上書きは不要どころか、**既存の正しいポリシーを壊すリスク**がある。

---

### 3. **CloudFormation Outputの不整合**

**実装手順書（正しい）：**
- Output一覧に以下が明記：
  - `CatalogRoleArn`, `WorkspaceRoleArn`, `S3BucketName`, `VPCId`, `PrivateSubnet1Id`, `PrivateSubnet2Id`, `DatabricksComputeSGId`, `RDSEndpoint`

**Gemini版（誤り）：**
- Outputとして `DatabricksUnityRoleArn` という**実装手順書に存在しないキー名**を使用
- `PrivateSubnet2Id` がOutputにないと主張し、「AWSコンソールから手動で探してください」と案内（Step 2-2③）
  - 「ただのバックアップサーバー用のサブネット。CloudFormationで指定を省略した。」という記述は設計意図の誤解

> **問題点**: CloudFormationが両方のPrivate SubnetのIDを出力するなら、手動で探す必要はない。

---

### 4. **Storage設定のIAMロールARN**

**実装手順書（正しい）：**
- Storage設定には **`CatalogRoleArn`**（Unity Catalog S3アクセス用）を指定

**Gemini版（誤り）：**
- Storage設定に **`northwind-databricks-workspace-role`のARN**（手動作成したワークスペースロール）を指定するよう案内

> **問題点**: ワークスペースロール（EC2管理用）とカタログロール（S3アクセス用）は**目的が異なる**。Storage設定にワークスペースロールを入れるのは権限設計として誤り。

---

### 5. **Storage Credential作成時のロールARN**

**実装手順書（正しい）：**
- Storage Credential には **`CatalogRoleArn`** を使用

**Gemini版（誤り）：**
- `DatabricksUnityRoleArn` という名前を使っているが、これが実際にどちらのロールを指すのか曖昧
- CloudFormation Outputのキー名と一致しない

## 🔧 SKILLファイルについて

共有いただいた `SKILL.md` は **Databricks CLIの操作リファレンス**（認証、プロファイル選択、データ探索、バンドルデプロイ）です。

**今回の文脈での有用性：**
- Phase 2-6のSecrets設定でCLIを使う場合 → `databricks auth profiles` で認証確認に使える
- 構築後のデータ探索 → `databricks experimental aitools tools query` が便利
- **ただし、ワークスペース作成やCloud Resources登録はCLIではなくAccount Console UIで行うため、今回のPhase 1-2の構築作業には直接関係しない**

---

## 最終結論

**実装手順書が正しい設計思想：**
> 「CloudFormationで必要なものをすべて作り、DatabricksのUIではOutputの値を貼り付けるだけ」

**Gemini版の根本的な誤り：**
> 「CloudFormationで作ったものを無視して、手動で同等のリソースを別途作らせている」

これにより、IAMロールの二重作成、バケットポリシーの上書き事故、ロールの取り違えなど、**本番環境では致命的な問題**につながります。
