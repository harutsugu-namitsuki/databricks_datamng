操作トレース可視化システム 調査報告

追加要件（反復再生）の before/after、「3秒で消える」不具合の真因、そしてマップ網羅性の真因調査と横並び点検。— 一度読んで腑に落ちることを狙った人間向け資料。

更新（2026-06-18）：本報告の③④はすべて対応済み。 推奨の案A（完全カバレッジ）を実装した。地図は nodes 21→41 / edges 24→54 / routes 5→16 / pages 5→10 に拡張し、誤ラベルは「メソッド+パス」照合で解消、employees/shippers テーブルと商品画像配信コンポーネント（静的配信→画像ファイル）を計測対象化。以下の表は「修正前」の棚卸しスナップショット（経緯の記録）として残す。
TL;DR（結論を先に）
  1. 反復再生は実装済み。経路は「1回流れて消える」から「次の操作が来るまで粒子の波を繰り返す」に変わった。
  2. 「3秒で消える」の真因はブラウザのキャッシュ。コードの3秒消灯タイマーは撤去済みだが、index.html にキャッシュバストが無く旧 viewer.js が再利用されていた。→ ?v= 付与＋サーバー側 no-store 配信で恒久対処。要・ハード再読み込み。
  3. マップは「アプリ全量」ではなく、モックアップの代表5フローを写した teaching subset。画面・エンドポイント・テーブルの大半が未登録。「従業員管理を押すと画面もハンドラも出ずfetchから」はこれが原因。
  4. 商品画像の配信はそもそも計測対象外。<img> のリソース読込は fetch を通らず、計測は /api のみのため、画像配信コンポーネントが地図に存在しない。

① リアルタイム経路アニメ：反復再生（before / after）

1操作（1トレース）が起こす経路を点灯し粒子を流す挙動を、次の操作まで繰り返すよう変更した。

Before

  • span受信時に粒子を1回だけ放出
  • setTimeout(…, 3000) で3秒後に消灯
  • 操作後しばらくすると地図が暗転

After

  • currentPath に最新経路を保持
  • loop() が WAVE_PERIOD(=1200ms) ごとに粒子の波を再放出
  • 3秒消灯タイマーを撤去。次トレース到着で最新経路に置換、履歴モードで停止

実体：src/web/trace/viewer.js の emitWave() / loop() / lightTrace()。変更履歴は自動実装記録 §9。

② 「3秒でアニメーションが消える」の真因

コード側の3秒消灯（lightTrace 内の setTimeout(…3000)）はすでに撤去済みで、現行 viewer.js には存在しない（残る 3000 は無関係な pruneTraces のTTLと resizeFx のみ）。それでも「3秒で消える」が続いていた。

真因：ブラウザが旧 viewer.js をキャッシュ再利用

index.html は <script src="/static/trace/viewer.js"> をバージョン無しで読み込んでいた。FastAPI の StaticFiles は明示的キャッシュ制御を返さないため、ブラウザが旧JS（3秒消灯あり）をそのまま使い続け、サーバー上のファイルを更新しても画面が変わらなかった。

恒久対処（2点）：
  1. index.html に viewer.js?v=20260618-loop2 のキャッシュバストを付与。
  2. サーバーで /static/trace/ を Cache-Control: no-store 配信（以後の編集が再読み込みで必ず反映）。実体：src/api/main.py のトレースミドルウェア。
※反映にはサーバー再起動＋ブラウザのハード再読み込み（Ctrl+F5）が必要。

③ マップ網羅性の真因と横並び点検

真因（1文）： 経路マップと対応表（trace_map.json の nodes/routes/pages）は、UIモックアップの代表5フロー（catalog/dashboard/login/inventory/orders）を写した teaching subset であり、アプリ全量の棚卸しをしていない。加えて計測は /api のみ・fetch のみが対象で、静的配信（<img> 等）は設計上トレースされない。

商品画像コンポーネントの欠落

EC を開くと catalog.html が <img src="/static/img/products/{id}.jpg"> で画像を直接読み込む。これは下図の通り API を経由しない静的配信であり、現状の計測では一切見えない。

ブラウザ <img> ──HTTP(GET /static/img/...)──▶ FastAPI StaticFiles(/static) ──▶ ファイルシステム src/web/img/products/ （= 本来あるべき「画像配信」コンポーネント）

計測されない2つの理由：

  1. 計測ミドルウェアは path.startswith("/api") のみ span 化 → /static/img/... は記録対象外。
  2. フロント計装 trace-client.js は window.fetch だけをパッチ → <img> のリソース読込は fetch を通らず trace_id すら付かない。

結果、「ECを開いた瞬間に画像をファイルから引っ張る」という実際の挙動が、地図にもリアルタイムにも現れない。ご指摘の通り本来コンポーネントとして存在すべき。

④ 推奨対応

マップを「教育用最小」から「アプリ実体に追従」へ引き上げる。粒度の選択肢：

案内容労力網羅性
A. 完全カバレッジ（推奨）全画面を pages に、全エンドポイントを routes＋ハンドラノードに、employees/shippers をテーブルノードに追加。ルート照合を「メソッド+パス」にして誤ラベル解消。画像配信を「静的配信」コンポーネントとして地図に追加し、計測も対応（下記）。中〜大高
C. 中間画面・ハンドラ・テーブルは全追加。画像配信は地図に静的表示（非点灯）に留め、計測は将来対応。中中〜高
B. 現状維持＋明示代表5フロー固定と割り切り、未収録は「地図外」とUIに注記。小低
推奨：A（または時間制約があれば C）。 画像配信を「実際に光る」コンポーネントにするには、trace-client.js で画像読込も検知（例：<img> 生成監視や PerformanceObserver で resource 取得を捕捉し X-Trace-Id 相当を擬似発火）するか、サーバー側で /static/img 配信も span 化する、のいずれか。サーバー側 span 化が最も確実（fetch でなくとも記録できる）。

※ どの案で進めるかは設計判断のため、着手前に確認します（ご指摘の「設計書にも反映」方針に沿い、確定後 要件定義書 と trace_map.json の双方を更新します）。
