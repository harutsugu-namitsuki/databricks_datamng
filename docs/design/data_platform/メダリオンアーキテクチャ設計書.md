# メダリオンアーキテクチャ設計書（AWSシングルクラウド版）

## 1. アーキテクチャ概要

### 1.1 メダリオンアーキテクチャとは

データレイクハウス上でデータの品質を段階的に高めるレイヤー設計パターン。
本プロジェクトでは **Bronze → Silver → Gold** の3層に加え、運用ログ用の **Ops** 層を設け、計4層構成とする。

### 1.2 採用理由

| 観点 | 採用理由 |
|------|---------|
| データ品質の段階管理 | 生データ（Bronze）と整形済みデータ（Gold）を分離し、品質問題の切り分けを容易にする |
| 再処理の容易さ | Bronze に原本を保持するため、変換ロジック変更時にソース再取得なしで再処理可能 |
| アクセス制御の明確化 | レイヤー＝スキーマ とすることで、Unity Catalog の GRANT 単位と一致させ権限管理を簡素化 |
| 運用の可視化 | Ops 層にログ・DQ結果を集約し、パイプラインの健全性を一元的に監視可能とする |

### 1.3 全体像

```
AWS RDS PostgreSQL (Northwind)
    │
    │  JDBC（Private Subnet内通信）
    ▼
┌─────────────────────────────────────────────────────────────────┐
│  Databricks (Unity Catalog: northwind_catalog)                  │
│                                                                 │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐               │
│  │  Bronze   │────▶│  Silver   │────▶│   Gold   │               │
│  │ (生データ) │     │ (標準化)  │     │ (集計)   │               │
│  └──────────┘     └──────────┘     └──────────┘               │
│       │                │                                        │
│       │                ▼                                        │
│       │          ┌──────────┐                                   │
│       └─────────▶│   Ops    │                                   │
│                  │ (運用ログ) │                                   │
│                  └──────────┘                                   │
└─────────────────────────────────────────────────────────────────┘
    │
    ▼
S3 (Delta Lake): s3://lake-northwind-<account-id>/
```

## 2. レイヤー設計方針

### 2.1 Bronze層（生データ）

| 項目 | 方針 |
|------|------|
| 目的 | ソースシステムの忠実な複製を保持する「原本」 |
| 対象 | RDS Northwind 全14テーブル |
| 保存形式 | Delta Lake |
| 更新方式 | Overwrite（日次実行ごとに当日分を冪等に書き込み） |
| メタデータ | `_run_id`, `_load_date`, `_ingest_ts`, `_source_system` の4列を付与 |
| データ加工 | 一切行わない（ソースのカラム名・型をそのまま保持） |
| 保持ポリシー | Delta Lake の Time Travel により過去バージョンを保持（デフォルト30日） |
| アクセス範囲 | Admin / DataEngineer のみ（個人情報混入リスクのため Analyst には非公開） |

**更新方式の選択根拠:** Northwind はテーブルサイズが小さく（最大 ~2,155行）、CDC（変更データキャプチャ）の導入コストに見合わないため、日次 Full Overwrite を採用。冪等性により再実行時も安全。

### 2.2 Silver層（標準化）

| 項目 | 方針 |
|------|------|
| 目的 | 型変換・NULL処理・重複排除を行い、分析に耐える品質を確保する |
| 対象 | Bronze 14テーブルのうち、Gold 集計に必要な4テーブル（customers, orders, order_details, products） |
| 保存形式 | Delta Lake |
| 更新方式 | Overwrite |
| メタデータ | Bronze の `_run_id` 等は除去（Ops 層で追跡可能なため冗長） |
| アクセス範囲 | Admin / DataEngineer のみ |

**Silver 対象テーブルの選定基準:** Gold マートの構築に必要なテーブルのみを Silver 化する。マスタ系テーブル（categories, shippers 等）は Bronze のまま参照する場合がある。

#### クレンジングルール

| ルール | 適用内容 | 対象テーブル |
|--------|---------|------------|
| NULL→デフォルト値 | 数値は `0`、文字列は `""` または `"N/A"` | 全テーブル |
| 型変換 | VARCHAR → DATE/TIMESTAMP、FLOAT → DECIMAL(10,2) | orders, order_details, products |
| 文字列正規化 | `trim()` + `upper()`（国名等） | customers, orders |
| 重複排除 | 主キーによる `dropDuplicates` | 全テーブル |
| 計算列追加 | `line_total = unit_price * quantity * (1 - discount)` | order_details |

### 2.3 Gold層（ビジネスマート）

| 項目 | 方針 |
|------|------|
| 目的 | ビジネスユーザー・ダッシュボードに直接提供する集計済みマートを作成する |
| 保存形式 | Delta Lake |
| 更新方式 | Overwrite |
| 粒度 | テーブルごとに定義（下表参照） |
| アクセス範囲 | Admin / DataEngineer（全権限）、Analyst（SELECT のみ） |

#### マートテーブル定義

| テーブル名 | 粒度 | 集計元（Silver/Bronze） | 主な集計指標 |
|-----------|------|----------------------|------------|
| sales_by_product | 商品 x 年月 | order_details + products + orders | total_sales, total_quantity, order_count |
| sales_by_customer | 顧客 x 年月 | order_details + orders + customers | total_sales, order_count, avg_order_value |
| sales_by_category | カテゴリ x 年月 | order_details + products + orders + categories(Bronze) | total_sales, total_quantity, order_count |
| order_summary | 日次 | orders + order_details | line_count, total_sales, avg_line_value |

**更新方式の選択根拠:** 集計マートは差分更新よりも全件再計算のほうがロジックが単純で、データ量も小さいため Overwrite を採用。

### 2.4 Ops層（運用ログ）

| 項目 | 方針 |
|------|------|
| 目的 | パイプラインの実行状況・データ品質を記録し、運用監視の基盤とする |
| 保存形式 | Delta Lake |
| 更新方式 | Append（ログの性質上、履歴は蓄積のみ） |
| アクセス範囲 | Admin / DataEngineer（全権限）、Analyst（SELECT のみ） |

#### 運用テーブル

| テーブル名 | 記録タイミング | 記録元ノートブック |
|-----------|-------------|----------------|
| job_runs | Gold 集計完了時 | `04_etl_gold_aggregate.py` |
| ingestion_log | Bronze 取り込み完了時（テーブル単位） | `02_etl_bronze_ingest.py` |
| dq_results | Silver 変換後のDQチェック完了時 | `03_etl_silver_transform.py` |

## 3. データフロー設計

### 3.1 パイプライン全体フロー

```
[RDS PostgreSQL]
      │
      │ (1) JDBC読み込み + メタデータ付与
      ▼
[Bronze層] ── ingestion_log ──▶ [Ops層]
 全14テーブル
      │
      │ (2) クレンジング・型変換・重複排除
      ▼
[Silver層] ── dq_results ────▶ [Ops層]
 4テーブル
      │
      │ (3) JOIN・集計
      ▼
[Gold層]  ── job_runs ───────▶ [Ops層]
 4マート
```

### 3.2 ノートブック対応表

| 実行順 | ノートブック | 対象レイヤー | 実行タイミング | 主な処理 |
|--------|-----------|-----------|-------------|---------|
| 0 | `00_setup_unity_catalog.py` | （環境構築） | 初回のみ | External Location・スキーマ作成 |
| 1 | `01_load_northwind_to_rds.py` | （ソース準備） | 初回のみ | Northwind SQL → RDS 投入 |
| 2 | `02_etl_bronze_ingest.py` | Bronze + Ops | 日次 | JDBC取り込み、ingestion_log 記録 |
| 3 | `03_etl_silver_transform.py` | Silver + Ops | 日次 | クレンジング、DQチェック、dq_results 記録 |
| 4 | `04_etl_gold_aggregate.py` | Gold + Ops | 日次 | 4マート集計、job_runs 記録 |

### 3.3 冪等性の確保

| レイヤー | 冪等性手段 | 説明 |
|---------|----------|------|
| Bronze | `mode("overwrite")` + `saveAsTable` | 同日の再実行で重複データが生じない |
| Silver | `mode("overwrite")` + `dropDuplicates` | 常に最新のBronzeデータから再構築 |
| Gold | `mode("overwrite")` | Silver から全件再集計 |
| Ops | `mode("append")` | ログは追記のみ（再実行時は複数レコードが残る） |

## 4. データ品質（DQ）チェック設計

### 4.1 設計方針

- DQ チェックは **Silver 層の書き込み直後** に実行する（Bronze は原本のため検証対象外）
- チェック結果は `ops.dq_results` に記録し、PASS/FAIL を判定
- 現時点では FAIL 時もパイプラインは停止せず、ログに記録するのみ（将来的にアラート通知を追加予定）

### 4.2 チェックルール一覧

| ルールID | ルール名 | 対象テーブル | チェック内容 | 閾値（fail_count） |
|---------|---------|-----------|-----------|-----------------|
| DQ-001 | null_check_company_name | silver.customers | `company_name` が NULL の件数 | 0（1件でもあればFAIL） |
| DQ-002 | dup_check_order_id | silver.orders | `order_id` の重複件数 | 0 |
| DQ-003 | positive_unit_price | silver.order_details | `unit_price < 0` の件数 | 0 |

### 4.3 今後の拡張方針

| 拡張項目 | 想定スプリント |
|---------|-------------|
| FAIL 時の Workflow 停止 | Workflow 自動化スプリント |
| CloudWatch + SNS によるアラート通知 | 運用強化スプリント |
| チェックルールの追加（参照整合性、範囲チェック等） | 随時 |
