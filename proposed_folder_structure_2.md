# Databricksデータマネジメントプロジェクト フォルダ構成再設計提案書 v2

本資料は、現在の「時期（フェーズ）」をベースとしたフォルダ構成を、直感的かつメンテナンス性の高い「役割（コード・ドキュメント・インフラ）」ベースのフォルダ構成に再整理するための提案書です。

> [!NOTE]
> ユーザー様からの指示に基づき、本提案に伴うファイル・フォルダの実際の移動や修正は一切実施しておりません。

> [!IMPORTANT]
> **v2での主な変更点**：レビュー指摘を反映し、(a) ファイルのリネームを移動から分離、(b) `git mv` 使用と参照パス更新工程を明記、(c) 移行を3フェーズに分割、(d) skills/ の移動を「要確認」扱いに変更、(e) 重複ファイル・生成物の取り扱いを明示しました。
> **v2.1追記**：(A) `cloudformation lambda.yaml` を原名（スペース入り）のまま維持するよう修正（「リネームしない」原則との整合）、(B) フェーズ2の参照更新対象を「リポジトリ相対パスのみ」に限定し、S3/DBFS等の実行時パスを明確に対象外化（稼働コード破壊の防止）。

## 1. 現状の構成（Before）における課題

1. **時期（フェーズ）によるトップレベルの分断**
   - `（～4月）AWSシングルクラウドVer` と `（5月～）APP開発` という時期別のフォルダがトップレベルに並列配置されており、それぞれに個別の `docs` や `README.md`、`src` が散在しています。これにより、プロジェクトの全体像が把握しづらくなっています。日付プレフィックスは時間経過とともに意味が腐るため、早期是正の価値があります。
2. **ドキュメント配下へのコード・IaCの混在**
   - `docs/03_development/...` の下に `cloudformation.yaml` や `notebooks/` (Databricks用PySparkコード) が、`docs/04_operation` の下にサンプルデータ出力用Pythonスクリプトが混在しており、「ドキュメント」と「実行コード」の境界線が曖昧です。
3. **リポジトリ全体の俯瞰用 README の欠如**
   - リポジトリのルート直後に時期別フォルダが配置されているため、初見でどのファイルから参照すべきかが分かりづらくなっています。
4. **CLAUDE.md の記述と実態の乖離**
   - CLAUDE.md は `マルチクラウドVer/` や `開発ドキュメント/notebooks/` を参照していますが、実際のフォルダ名は `docs/03_development/...` であり、マルチクラウド版は存在しません。今回の再編はこのズレを一掃する好機です。
5. **参考資料（skills/）のトップレベル配置**
   - プロジェクトコードとは直接関係のないClaude向け参考スキル群がトップレベルに置かれており、ディレクトリの認知的負荷を高めています。（※ただし移動可否は要検証。後述）

## 2. 提案するフォルダ構成（After）の設計方針

1. **「コード (src/databricks/infrastructure)」と「ドキュメント (docs/)」の完全分離**
   - 時期別という括りを排除し、ファイルの性質（アプリケーションコード、データ処理コード、IaC、設計・運用ドキュメント）ごとに一元管理します。
2. **Databricks関連コードの集約**
   - Databricks上で動かすNotebookや、運用スクリプトなどのデータ系コードを `databricks/` ディレクトリに集約します。
3. **設計ドキュメントのカテゴリ別統合**
   - AWSシングルクラウドのデータプラットフォーム設計とAPP開発のシステム・画面設計を `docs/design/` の中でカテゴリ分けして整理し、システム全体のアーキテクチャを一箇所で俯瞰できるようにします。
4. **トップレベル README の新設**
   - リポジトリのトップレベルに総合 `README.md` を作成し、プロジェクト概要や新しいフォルダ構造の案内を記述します。
5. **【v2追加】フォルダ名は英語化するが、ファイル名は日本語のまま維持**
   - 本リポジトリは中身が日本語であり、日本語ファイル名は検索性が高いため、ファイル名の英語化は行いません。リネームは相互参照リンクを大量に破壊するリスクがあるため、移動とは切り離して扱います。

## 5.【v2新設】移行手順（フェーズ分割）

問題発生時にどの工程で壊れたか切り分けられるよう、移行を3フェーズに分割します。

### フェーズ1：移動のみ（1コミット）

- **すべて `git mv` で実施**します。エクスプローラやOneDrive同期によるドラッグ移動は、Gitが「削除＋新規追加」と見なし `git log --follow` での履歴追跡が切れるため厳禁です。
- **OneDrive配下のため、移行作業中は同期を一時停止**し、同期競合を防止します。
- **この段階ではファイル名のリネームは一切行いません**（日本語ファイル名を維持）。
- 移動対象は本書マッピング表のとおり。`skills/` は次項の確認が済むまで保留。

### フェーズ2：参照パスの一括更新（1コミット）

移動によって壊れる内部参照（クロスリンク）を一括修正します。

- **修正対象（＝リポジトリ内の相対パス参照のみ）**：
  1. `CLAUDE.md` の Key Files テーブル・Notebook実行順（＋ 既存の現状ズレ：`マルチクラウドVer/`・`開発ドキュメント/notebooks/` 等も是正）
  2. 各手順書（`実装手順書.md` 等）が参照する `notebooks/`・`cloudformation.yaml` 等の**リポジトリ相対パス**
  3. 設計書・手順書間の相互リンク（Markdownリンク）
- **【重要】対象外（触らない）**：Notebook内の `dbutils`／S3キー、CloudFormation内のS3バケット・キー等の**実行時（S3/DBFS）パス**は、リポジトリのフォルダ移動とは無関係であり、書き換えると稼働中のコードを壊す。機械的なgrep置換の対象に含めない。
- **工程**：移動後に「リポジトリ相対パスとして」`notebooks/`・`cloudformation` 等の参照を検索し、ドキュメント側のリンクのみ新パスへ更新。実行時パスはヒットしても変更しない。

### フェーズ3：任意・後日対応

- **skills/ の移動可否確認**：`.claude/` 等の規約パスからの参照有無を `grep` で検証。参照依存がある場合は移動せず据え置き、または `.gitignore`／リポジトリ外管理を検討。
- **重複ファイルの棚卸し**：`cloudformation.txt`・`cloudformation lambda copy.txt` 等について、残す/削除を明示して整理。
- **生成物の扱い決定**：`サンプルデータ統合_*.xlsx` を Git管理継続／`archive/`／`.gitignore` のいずれにするか決定。
- **（任意）ファイル名の英語化**：メリットが薄くリスクが高いため、原則保留。実施する場合も本フェーズで単独コミット。
