## 📌 Epic 6: 業務アプリ＋購買アプリの構築とデータレイクハウス統合

*目的: 業務側アプリ（管理画面）とユーザー側アプリ（購買画面）の2つを構築し、アプリ操作がパイプライン→ダッシュボードに反映されることを一気通貫で実証する*

| Story | 内容 |
|-------|------|
| **業務側アプリ（管理画面）** | |
| 6-1 | Streamlit等で業務管理アプリの骨格を作成する（ログイン画面、メニュー画面） |
| 6-2 | 商品マスタの登録・編集機能を実装する |
| 6-3 | 在庫管理機能を実装する（入荷登録、在庫数の手動調整） |
| 6-4 | 従業員管理機能を実装する（従業員情報の閲覧・編集、権限に応じたマスキング表示） |
| 6-5 | 業務アプリの入力バリデーションを実装する（必須項目チェック、数値範囲チェック等） |
| **ユーザー側アプリ（購買画面）** | |
| 6-6 | Streamlit等でユーザー向け購買アプリの骨格を作成する（商品一覧画面） |
| 6-7 | 商品詳細表示・カート追加機能を実装する |
| 6-8 | カート画面・注文確定機能を実装する（RDSのorders/order_detailsテーブルにINSERT） |
| 6-9 | 注文履歴機能を実装する（過去の注文一覧と詳細の表示） |
| 6-10 | 購買アプリの入力バリデーションを実装する（在庫チェック、数量チェック等） |
| **データ連携の実証** | |
| 6-11 | 業務アプリで商品を新規登録 → 日次パイプライン実行 → Gold層に反映 → ダッシュボードに新商品が表示されることを確認する |
| 6-12 | 購買アプリで注文を実行 → 日次パイプライン実行 → Gold層の売上集計に反映 → ダッシュボードの売上グラフが更新されることを確認する |
| 6-13 | 購買アプリで大量注文 → 在庫が発注点を下回る → ダッシュボードの在庫アラートが発火することを確認する |
| **コスト最適化** | |
| 6-14 | RDSの起動/停止をDatabricks Job前後で自動化する（Lambda + EventBridge、またはDatabricks Job内からAWS API呼び出し） |

## 📌 Epic 7（アプリ関連）: アプリに関するテスト戦略の実装

*目的: Epic 6で構築したアプリに対して、単体テスト・結合テスト・E2Eテストを実装する*
*※ パイプラインの単体テスト・結合テスト等はメインバックログのEpic 7に残置*

| Story | 内容 |
|-------|------|
| **単体テスト（スタブ使用）** | |
| 7-3 | 業務アプリのRDS書き込みロジックの単体テストを作成する。実際のRDSではなく**スタブ（モックDB）**を使い、INSERT文が正しく生成されるかassertする |
| 7-4 | 購買アプリの在庫チェックロジックの単体テストを作成する。在庫0の商品に対する注文が正しくエラーになるかassertする |
| **結合テスト（一部スタブ）** | |
| 7-7 | アプリ→RDS書き込み→Bronze取り込みの結合テスト。アプリ部分は本物、パイプライン部分も本物だが、**テスト専用のRDSスキーマ**を使って本番データを汚さない |
| **E2Eテスト（スタブなし・全部本物）** | |
| 7-8 | 購買アプリで注文 → RDS更新 → パイプライン実行 → Gold層の集計結果が期待値と一致 → ダッシュボードに反映。**全て本物のシステムを使う** |
| 7-9 | 業務アプリで商品登録 → RDS更新 → パイプライン実行 → Gold層に反映 → ダッシュボードに表示。**全て本物** |

## 📌 Epic 9（アプリ関連）: アプリのCI/CDとデプロイ自動化

*目的: アプリをコンテナ化してAWS上にデプロイし、コード変更が自動でテスト・デプロイされるCI/CDパイプラインを構築する*
*※ メインバックログのEpic 9（Databricks基盤のCI/CD）とは独立したアプリ専用のCI/CD*

| Story | 内容 |
|-------|------|
| **コンテナ化** | |
| 9-A1 | 業務アプリ・購買アプリのDockerfile を作成し、docker-compose でローカル環境での動作を確認する |
| **デプロイ環境構築** | |
| 9-A2 | AWSにアプリのホスティング環境を構築する（App Runner または ECS Fargate）。CloudFormation/Terraformでインフラをコード化する |
| **CI/CDパイプライン** | |
| 9-A3 | GitHub Actionsで、mainブランチへのpush時にコンテナビルド→AWSへの自動デプロイを行うCI/CDパイプラインを構築する |
| 9-A4 | CI/CDパイプラインにアプリ単体テスト（7-3, 7-4）を組み込み、**テストが通らないとデプロイされないゲート**を設ける |
