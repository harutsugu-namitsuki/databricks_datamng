### ⚠️ 重大な注意点・リスク



#### 🚨 リスク1: **外部ロケーションの重複（Overlap）問題**



これが**最も重要な懸念点**です。



画像を見ると、既に以下の外部ロケーションが存在しています：



| 外部ロケーション | URL |
|---|---|
| ext_bronze | `s3://lake-northwind-784318224989/bronze` |
| ext_gold | `s3://lake-northwind-784318224989/gold` |
| ext_ops | `s3://lake-northwind-784318224989/ops` |
| ext_silver | `s3://lake-northwind-784318224989/silver` |
| northwind_workspace_74746... | `s3://lake-northwind-784318224989/unity-catalog/7474655632...` |



手順では、新たに `s3://lake-northwind-784318224989/`（バケットルート）を外部ロケーション `ext_northwind_root` として作成しようとしています。



**公式ドキュメントには明確に以下の制約が記載されています：**



> **External locations can't overlap other external locations.**
> 
> **You can't define an external location within another external location.**



- 参照: [Path rules and access in Unity Catalog volumes](https://docs.databricks.com/aws/en/volumes/paths)



つまり、**バケットルート `s3://lake-northwind-784318224989/` は、既存の `ext_bronze`（`s3://lake-northwind-784318224989/bronze`）等の親パスにあたるため、重複（overlap）として作成がエラーになる可能性が高い**です。



#### 🚨 リスク2: **ワークスペースデフォルトストレージとの競合**



画像の5番目のエントリ `northwind_workspace_74746...` は、ワークスペースのデフォルトUnity Catalogストレージ（`s3://lake-northwind-784318224989/unity-catalog/...`）を指しています。



公式ドキュメントには以下の警告があります：



> **A storage path conflict occurs when an external location overlaps with the internal path Databricks uses for your workspace's default Unity Catalog storage.** This overlap interferes with how Unity Catalog uses the workspace's default storage for internal processing in certain features.
> 
> **This also weakens data governance because the external location exposes the workspace internal data inside the location.**



- 参照: [Resolve storage path conflicts](https://docs.databricks.com/aws/en/data-governance/unity-catalog/storage-conflicts)



バケットルート全体を外部ロケーションにすると、ワークスペースの内部ストレージパスも包含してしまい、**ガバナンスの弱体化やUnity Catalogの一部機能がブロックされるリスク**があります。



#### ⚠️ リスク3: **ベストプラクティスとの乖離**



Databricks公式のベストプラクティスでは：



> **Avoid path overlap conflicts: never create external volumes or tables at the root of an external location.**



- 参照: [Unity Catalog best practices](https://docs.databricks.com/aws/en/data-governance/unity-catalog/best-practices)



バケットルート全体を外部ロケーションにすること自体が推奨されていません。

### 💡 推奨される代替アプローチ



既に `ext_bronze`, `ext_gold`, `ext_silver`, `ext_ops` が個別に存在している状況を踏まえると、以下のいずれかが適切です：



#### 案A: カタログ用の専用サブディレクトリを作成する（推奨）



```
External location name: ext_northwind_catalog
URL: s3://lake-northwind-784318224989/catalog/
Storage credential: aws_s3_credential
```



```sql
CREATE CATALOG IF NOT EXISTS northwind_catalog
MANAGED LOCATION 's3://lake-northwind-784318224989/catalog/';
```



既存の `bronze/`, `gold/`, `silver/`, `ops/`, `unity-catalog/` と重複しない新しいサブディレクトリを指定します。



#### 案B: 別のS3バケットを用意する



カタログのマネージドストレージ用に専用のS3バケットを新規作成し、そこを外部ロケーションとして登録する方法です。これが最もクリーンな分離になります。
