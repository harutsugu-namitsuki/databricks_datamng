## まず3つの言葉をかんたんに

| 用語 | たとえ | 特徴 |
|------|--------|------|
| **データレイク** | 巨大な「倉庫」 | なんでもとりあえず放り込める。整理はされていない |
| **データウェアハウス (DWH)** | きれいに整理された「図書館」 | 検索しやすく、すぐ分析に使える。ただし入れる前に整理が必要 |
| **データレイクハウス** | 倉庫と図書館が**合体**したもの | 放り込む自由さ＋検索・分析の速さを両立 |

## この開発のどこが何にあたるか

全体の流れを図にすると：

```
RDS PostgreSQL（業務DB）
    │
    │  01_load_northwind_to_rds（データ投入）
    ▼
┌─────────────────────────────────────────────┐
│           S3 バケット（Delta Lake）            │
│                                             │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │ Bronze  │→│ Silver  │→│  Gold   │     │
│  │(生データ)│  │(整理済み)│  │(集計済み)│     │
│  └─────────┘  └─────────┘  └─────────┘     │
│                                             │
│  ← これ全体が「データレイクハウス」 →          │
└─────────────────────────────────────────────┘
         ▲
    Databricks + Unity Catalog が管理
```

### 1. データレイク部分：**S3バケット ＋ Bronze層**

```
場所：S3 (lake-northwind-xxxx)
ファイル：cloudformation.txt の DataLakeBucket
コード：02_etl_bronze_ingest.py
```

- S3はただの**巨大な箱**です。ファイルを何でも置けます
- Bronze層は「RDSから取ってきたデータを**そのまま**放り込む」場所
- メタデータ（`_run_id`, `_load_date`など）は付けますが、**データ自体は加工しない**
- → これが**データレイク的な役割**（「とりあえず全部保存」）

### 2. データウェアハウス的な部分：**Silver層 ＋ Gold層**

```
コード：03_etl_silver_transform.py → 04_etl_gold_aggregate.py
```

**Silver層**（整理棚）：
- 文字列のトリム、NULL処理、型変換
- `line_total`（売上金額）の計算
- 品質チェック（DQ Check）
- → 「使える状態にきれいにする」

**Gold層**（分析レポート）：
- `sales_by_product`（商品別売上）
- `sales_by_customer`（顧客別売上）
- `sales_by_category`（カテゴリ別売上）
- `order_summary`（日次受注サマリ）
- → 「経営者やアナリストがすぐ見られる集計テーブル」

この Silver → Gold の部分は、従来の**データウェアハウスがやっていたこと**と同じです。

### 3. データレイクハウス：**この仕組み全体**

ここが一番大事なポイントです。

| 従来の方法 | この開発の方法 |
|-----------|--------------|
| S3（レイク）とRedshift（DWH）を**別々に**用意 | S3**1つの中に**Bronze/Silver/Goldを全部置く |
| データを2箇所にコピーして管理が大変 | **Delta Lake形式**で保存するので、S3なのにDWHのように高速検索・更新できる |
| 権限管理がバラバラ | **Unity Catalog**で一元管理 |

つまり：

> **S3（安くて大容量）の上に、Delta Lake（高速・信頼性）を載せ、Unity Catalog（権限管理）で統治する**
> 
> → これが「**データレイクハウス**」

## 各コンポーネントの役割まとめ

| コンポーネント | 役割 | レイク？DWH？レイクハウス？ |
|--------------|------|------------------------|
| **S3バケット** | データの物理的な保存先 | レイク（箱そのもの） |
| **Delta Lake形式** | S3上のファイルにトランザクション・バージョン管理を追加 | レイクをDWH級に進化させる技術 |
| **Bronze層** | 生データをそのまま保存 | レイク的 |
| **Silver層** | クレンジング・標準化 | DWH的 |
| **Gold層** | ビジネス向け集計 | DWH的 |
| **Unity Catalog** | テーブル・権限の一元管理 | レイクハウスの管理基盤 |
| **Databricks** | 上記すべてを動かすエンジン | レイクハウスのプラットフォーム |
| **RDS PostgreSQL** | 元の業務データベース（ソース） | どれでもない（データの出発点） |

---

## 一言でまとめると

> **この開発全体が「データレイクハウス」です。**
> 
> その中に、データレイク的な部分（S3 + Bronze）と、データウェアハウス的な部分（Silver + Gold）が**共存**しています。
> 
> 従来は別々のシステムだったものを、**Delta Lake + Unity Catalog + Databricks** で1つに統合したのが、この構成の本質です。
